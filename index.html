Customer Support AI Chatbot
RAG · NLP · Deployment

Identified a recurring bottleneck in a customer support team: agents were relying on memory and outdated documentation to answer questions, leading to slow, inconsistent responses. Built an end-to-end AI assistant — from raw audio to a live Telegram bot — entirely from scratch.

01
Speech-to-Text — Whisper
Transcribed years of customer service call recordings using OpenAI Whisper. Handled speaker diarization and filler word normalization at scale.
↓
02
Data Structuring & Chunking
Cleaned and tagged transcripts by issue type, resolution, and timeline. Chunked into semantically meaningful case summaries for retrieval.
↓
03
Vector Embeddings & Semantic Search
Generated embeddings from the structured corpus and indexed them in a vector database (FAISS / ChromaDB), enabling semantic retrieval over thousands of real past cases.
↓
04
RAG Architecture — LLM Layer
Built a Retrieval-Augmented Generation pipeline. Agent queries trigger context retrieval from the vector store; an LLM synthesizes grounded, accurate responses — no hallucinations, no static FAQs.
↓
05
Telegram Bot Deployment — Python
Deployed the full pipeline as a Telegram bot using Python (python-telegram-bot), giving support staff real-time access mid-conversation with zero new tooling required.
Years of call recordings → searchable knowledge base
Standardized answers across the support team
Cut new-hire ramp time by reusing real past cases
Built and deployed at zero external software cost
ATM Site Prospecting Tool
Google Places API · Mapping · CRM

Our business development process for finding new ATM host locations relied entirely on manual Google Maps searches — a platform that caps results at ~20 per view. The team had to zoom in, collect results, pan the map, and repeat across entire cities. It was slow, inconsistent, and guaranteed to miss locations. I built a web tool using AI that eliminated the manual search entirely, surfacing over 400 qualifying locations across key West Coast markets — replacing it with an automated grid-scanning system backed by a live map dashboard and built-in lead tracker.

01
Auto-Zoning & Grid Scan
The tool divides any target area into a systematic grid of zones and scans each one automatically via the Google Places API — bypassing the ~20 result cap that made manual search unscalable.
↓
02
Business Hours Filtering
Automatically filters results to only surface qualifying locations: open no later than 10am, close no earlier than 8pm, with 24-hour businesses flagged as priority targets — matching our ATM placement criteria exactly.
↓
03
Live Map with Location Pins
All qualifying businesses are rendered as color-coded pins on an interactive map, giving the team an instant visual overview of the entire prospect landscape across a city — something manual Google Maps searches could never provide.
↓
04
CRM Dashboard & Lead Tracking
A built-in dashboard tracks every location across three states: Potential (identified, not yet contacted), Contacted (outreach in progress), and Owned (ATM installed). Replaced scattered spreadsheets with a single source of truth for the BD team.
400+ qualifying locations surfaced across West Coast markets
Replaced manual map-by-map search workflow entirely
Single dashboard tracking prospecting status across all sites
Filters auto-match ATM placement criteria (hours, 24/7 flag)
Self-Hosted Video Dubbing & Transcription Suite
Multimodal · Local + Cloud · Multi-Interface

Foreign-language video content stays foreign — existing auto-dub tools either feel robotic or live behind paywalls. I built two open Telegram bots on one shared media-AI pipeline: Whisper_Bot (incoming audio/video → transcription + translation) and Dub_Bot (source video → fully dubbed version, optionally in the original speaker's voice). The interesting work isn't wiring up the models — it's the production decisions: when a fast cloud TTS beats a local voice clone, how to keep music intact when replacing speech, how to monitor and retrieve long-running jobs from a phone. Same codebase, two interfaces (Telegram + Gradio web UI), every trade-off load-tested by daily use.

01
Source Intake — yt-dlp + Direct Upload
Pulls source video from YouTube, Bilibili, or direct file upload via yt-dlp. Codec preference (H.264 over AV1) added after older iPhones silently failed to decode certain streams — the kind of edge case you only catch in production.
↓
02
Transcription — Whisper Large-v3
Whisper Large-v3 with VAD filtering and int8 quantization to keep CPU inference fast. One transcription model serves both bots — Whisper_Bot returns it directly, Dub_Bot pipes it into translation.
↓
03
Translation — DeepSeek with Timing Budgets
DeepSeek translates each line within a per-line word budget calculated from its subtitle slot — dubbed audio has to fit the original time window or it overflows the next subtitle. Three-tier fallback (TTS rate-adjust → LLM shortening → time-stretch) handles overruns without warping meaning.
↓
04
Voice Synthesis — Dual Backend
Two TTS backends picked per job: Edge-TTS (cloud, fast, generic voice) for routine dubs, or F5-TTS — a local voice-clone trained from a 10-second reference of the original speaker — when the dub needs to sound like that person. Demucs separates vocals beforehand so the new dub mixes over the original music with no speech bleed.
↓
05
Delivery — Telegram + Web GUI
Same pipeline served two ways: a Telegram bot for ops-from-anywhere — browse past jobs, fetch outputs, monitor progress all from a phone — and a Gradio web UI for batch processing at the desk. gofile.io fallback handles files above Telegram's 50 MB upload cap.
Two production Telegram bots from one shared media-AI codebase
Per-job toggle for cost-vs-fidelity (cloud TTS vs local voice clone)
Music-preserving dubs via vocal/instrumental separation
Remote ops from phone: status, project browse, file retrieval
